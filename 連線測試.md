# BIM Streaming 連線測試

用途：快速確認 `bim-streaming-server` 重新啟動後，`web-viewer-sample` client 是否可以連線並看到 WebRTC 串流畫面。

## 目錄

```text
server: C:\Repos\active\iot\AI-BIM-governance\bim-streaming-server
client: C:\Repos\active\iot\AI-BIM-governance\web-viewer-sample
```

## 1. 測試前先確認沒有殘留 server

在 PowerShell 執行：

```powershell
Get-Process kit -ErrorAction SilentlyContinue
netstat -ano | Select-String ':49100|:47998|:5173'
```

預期：

- 沒有舊的 `kit.exe`
- `49100` / `47998` 沒有被舊 server 占用
- 若 `5173` 已被占用，先確認是不是舊的 Vite client

## 2. 啟動 server

建議使用官方 headless streaming 路徑：

```powershell
cd C:\Repos\active\iot\AI-BIM-governance\bim-streaming-server
.\repo.bat launch -n ezplus.bim_review_stream_streaming.kit -- --no-window
```

或直接執行已 build 產物：

```powershell
cd C:\Repos\active\iot\AI-BIM-governance\bim-streaming-server
.\_build\windows-x86_64\release\ezplus.bim_review_stream_streaming.kit.bat --no-window
```

server log 預期看到：

```text
Started primary stream server on signal port 49100 and stream port 47998
app ready
```

## 3. 啟動 client

另開一個 PowerShell：

```powershell
cd C:\Repos\active\iot\AI-BIM-governance\web-viewer-sample
npm run dev -- --host 127.0.0.1
```

Vite 預期看到：

```text
Local:   http://127.0.0.1:5173/
```

## 4. Browser 操作

開啟：

```text
http://127.0.0.1:5173/
```

使用預設選項：


server log 預期看到：

## 6. 正確關閉順序

乾淨關閉建議順序：

1. 先關 browser viewer 頁面。
2. 在 client terminal 按 `Ctrl+C` 停 Vite。
3. 在 server terminal 按 `Ctrl+C` 停 Kit server。
4. 若 PowerShell 問 `Terminate batch job (Y/N)?`，輸入 `Y`。
5. 檢查 `kit.exe` 是否消失：

```powershell
Get-Process kit -ErrorAction SilentlyContinue
netstat -ano | Select-String ':49100|:47998|:5173'
```

若某次 server terminal 已關閉但 `kit.exe` 還在，先確認 PID 路徑是本 repo：

```powershell
Get-Process kit -ErrorAction SilentlyContinue | Select-Object Id,Path,StartTime
```

必要時精準停止該 PID：

```powershell
Stop-Process -Id <PID>
```

仍無法停止時才使用：

```powershell
Stop-Process -Id <PID> -Force
```

不要直接 `Stop-Process -Name kit -Force`，避免誤殺其他 Omniverse / Kit app。

## 7. 不乾淨關閉後的判斷

若沒有先關 browser client 就直接關 server，server log 可能出現：

```text
Failed to setup the streaming session because: StreamSdkException 800b0000 [NVST_R_GENERIC_ERROR] Got stop event while waiting for client connection.
```

這代表 WebRTC session setup 被 shutdown 打斷。它不一定代表 port 或 process 沒釋放，但會讓後續排查 log 變髒。

若重啟後 log 持續出現以下 pattern：

```text
Processing 13 signaling headers
Got stop event while waiting for client connection
```

處理方式：

1. 關掉所有 `127.0.0.1:5173` / `localhost:5173` 的 browser tab。
2. 停掉 Vite client。
3. 停掉 server。
4. 確認 `kit.exe` 和 `49100` / `47998` 已釋放。
5. 依本文件第 2-4 節重新啟動並按 `Next`。

## 8. Codex / 自動化測試注意事項

一般人工測試請優先使用 terminal 直接啟動，不需要 scheduled task。

在 Codex sandbox 內，直接 `Start-Process` Kit / npm 可能遇到 Windows session、GPU、Winsock 或 module loading 問題，不能代表一般桌面 session 的真實結果。若需要由 Codex 自動化啟動，較可靠的方式是用 Windows Scheduled Task 在互動桌面 session 執行。

這次踩到的自動化問題與解法：

- `Start-Process` 直接啟動 server/client 失敗，錯誤包含「目錄名稱無效」或「找不到指定的模組」。
- PowerShell `ScheduledTasks` cmdlet 在此 shell 不可用，改用 `schtasks.exe`。
- `schtasks.exe /TR` 直接塞 `powershell -Command "Set-Location ...; npm run dev"` 時，引號容易被吃壞，Vite 不會真的啟在 `5173`。
- 解法是用暫時 `.cmd` wrapper 進入 client repo 再執行 `npm run dev -- --host 127.0.0.1`。
- Chrome 的 `--remote-debugging-port` 在既有 Chrome instance/profile 下可能沒有開出 `DevToolsActivePort`；本次最後用 Windows UI Automation 找到 `Next` button 並 invoke，server log 隨後出現 `Client connected to WebRTC server`。

自動化驗證仍以 server log 和畫面為準，不要只看 `curl` / `netstat`，因為 Codex shell 內的 TCP 查詢曾出現與桌面實際狀態不一致。
